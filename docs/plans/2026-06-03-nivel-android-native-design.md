# Nivel — нативное Android-приложение тренера (Kotlin + Compose)

**Дата:** 2026-06-03
**Статус:** дизайн уточнён по результатам анализа кодовой базы; план на утверждении
**Охват:** **только сторона тренера.** Ученики остаются в вебе на тех же данных.

---

## Зачем

Боль тренера: после тренировки нет времени вручную заливать аудиозапись в веб. Нужно записывать прямо
на телефоне (в кармане, экран заблокирован, 60–90 мин) и чтобы запись автоматически уходила в
существующий конвейер «транскрипция → AI-инсайты».

Браузер этого надёжно не умеет (вкладка усыпляется при локе). Нативное Android-приложение с
**foreground service** (`microphone`) пишет в фоне штатно — это и есть причина строить нативку.

## Охват: тренерское приложение (решено)

Анализ кода показал: Nivel — это фактически **два приложения на общей базе** (тренер + ученик).
Сторона ученика (дашборд, vault принятых карточек, матчи/Playtomic, рефлексии) работает в вебе и боли
не вызывает. Поэтому нативно портируем **только сторону тренера**; ученик продолжает пользоваться
вебом. База и бэкенд общие — всё, что тренер делает в приложении, ученик сразу видит в вебе.

**В охвате (тренер):** ученики (создание теневых профилей + приглашения), тренировки (планирование,
упражнения, навыки), **запись аудио**, транскрипт, AI-разбор и ревью карточек, библиотека
карточек-шаблонов и коллекции, мастер-план ученика, справочники навыков/упражнений.

**Вне охвата (остаётся в вебе):** дашборд ученика, решения ученика по карточкам, vault, матчи/Playtomic,
рефлексии, просмотр мастер-плана учеником.

## Правила инициативы (обязательны)

1. **Никакого выдуманного UI.** Приложение воспроизводит существующий веб-интерфейс Nivel один-в-один.
   Разрешено только неизбежно-нативное (системные разрешения, уведомление записи в шторке, share-intent,
   нативные пикеры/навигация). Новые экраны, перекомпоновка, смена дизайна — нельзя. Сомнение = остановись
   и спроси. Эталон — соответствующий экран веба.
2. **Метка параллельности.** Лейбл `parallelizable` = задачу можно брать одновременно с другими (после
   закрытия зависимостей). Без лейбла — финально-интеграционные задачи (A6, C6, D5, G5), идут после соседей.
3. **Код-ревью перед сменой статуса.** Доделал → агент `code-reviewer` → правки по ревью → только потом PR
   и `in_review`. Смена статуса до завершённого ревью запрещена.

## Ключевой принцип: одна сущность, не две

Приложение и веб — **два клиента одного бэкенда** и **одной базы Supabase**
(`gqcyaxxhvyvpzuhoysis`). База — единственный источник правды. Бизнес-логика НЕ дублируется:
существующие server actions рефакторятся так, чтобы их ядро вызывалось и из web-actions, и из нового
REST API. Room в приложении — кэш чтения, не вторая база.

## Что показал анализ кода (факты)

**Авторизация:** вход через Гречку (OAuth → Firebase ID token → HMAC-JWT в куке `__session`, HS256,
`SESSION_SECRET`, 14 дней; `getSession`/`createSession` в `src/lib/auth/session.ts`; верификация
Firebase через JWKS в `src/lib/firebase/admin.ts`). Есть Google Sign-In fallback (`/api/auth/session`)
и claim-флоу приглашений (`claimSession`, `/invite/{token}`). Защита роутов — `src/proxy.ts`.

**Слой данных:** ~12 файлов server actions (`src/lib/actions/*`): sessions, audio, aiInsights,
insightCards (+ vault/collections/templates), goals, students (shadow+invite), skills, dashboard,
masterPlan, playtomic, library, telegram. Доступ к Supabase — через service role (`supabase/server.ts`,
обходит RLS); авторизация на уровне приложения (`requireTrainerOwnsSession` и т.п.). Две RPC:
`increment_skill_progress`, `replace_ai_draft_cards` — сохранить.

**Конвейер аудио:** `AudioUploader` → `requestAudioUploadUrl` (signed URL в bucket `session-audio`) →
PUT → `transcribeSession` (Groq `whisper-large-v3`, ru, `src/lib/stt/*`) → постобработка → запись в
`transcripts` → `generateAiInsights`/`pasteInsightsFromClaude` (OpenRouter, `src/lib/ai/*`) → draft
карточки → ревью тренером → решение ученика. ENV: `GROQ_API_KEY`, `OPENROUTER_API_KEY`, `INSIGHTS_MODEL`.

**Telegram:** webhook входящих (`/api/telegram/*`) + исходящие уведомления (`src/lib/telegram/notify.ts`):
новые инсайты, напоминание о тренировке, ученик завершил разбор. Пригодится; для нативки добавим FCM.

**Риски для нативки:** кука→bearer (нужен обмен токена), server actions→REST-обёртки, TTL signed URL
для длинной заливки, i18n (`Accept-Language`/`nameRu`/`nameEn`), сохранить RPC и ownership-проверки.

## Технические решения (приняты)

| Решение | Выбор | Почему |
|---|---|---|
| Клиент | Kotlin + Jetpack Compose | Нативный стандарт, полный доступ к фоновой записи |
| Репо | Отдельный `nivel-android` | Разные тулчейны; чистая HTTP-граница; своя доска |
| Граница | REST `/api/v1/*` (JSON) в репо NIVEL | Server actions нельзя звать из нативки |
| Логика | Рефактор ядра actions → общее для web и API | Ноль дублирования и расхождений |
| Auth | Firebase + обмен ID token → bearer-JWT (тот же HMAC). **НЕ** Supabase Auth | Сохраняем вход как есть, не трогаем веб |
| Запись | Foreground service `microphone` + AudioRecord/MediaRecorder (AAC/m4a) | Надёжно в кармане с локом; m4a уже в конвейере |
| Заливка | WorkManager + резюмируемая чанковая загрузка; увеличить TTL signed URL | Запись 90 мин не теряется при обрыве сети |
| Кэш | Room (кэш чтения); источник правды — Supabase | Скорость/оффлайн без второй базы |
| Пуши | FCM (Firebase уже есть) + существующий Telegram | Нативные пуши без новой инфраструктуры |
| Стек | Retrofit/OkHttp, Coroutines/Flow, Hilt, DataStore | Современный Android |

## UX-решения (от тренера)

- **Старт записи:** сначала выбрать ученика/тренировку, затем «Запись» (запись сразу привязана к сессии).

## Эпики и состав

### Бэкенд (репо NIVEL)

**Epic 0 — REST API `/api/v1` + bearer-авторизация (тренер-scoped)**
- A1 рефактор ядра server actions в переиспользуемые функции
- A2 bearer-авторизация: обмен Firebase ID token → JWT; `getSession` читает cookie|bearer
- A3 read-endpoints (тренер): ученики, профиль ученика (цели/сессии), карточка сессии, инсайт-карточки,
  справочники навыков/упражнений/проблем, мастер-план
- A4 audio-endpoints: upload-url (увеличенный TTL) + запуск транскрипции + статус
- A5 write-endpoints (тренер): создать/править тренировку и упражнения, цели для ученика,
  ученики+приглашения, ревью карточек (approve/reject/edit/reorder/review-complete),
  generate/paste инсайтов, коллекции/шаблоны, мастер-план, навыки
- A6 контракт-тесты + описание API

### Приложение (репо nivel-android)

**Epic 1 — каркас, вход, просмотр**
- B1 scaffold проекта (Compose, навигация, Hilt, сетевой слой Retrofit/OkHttp, DataStore)
- B2 вход: WebView-флоу Гречки + Google Sign-In fallback → сохранение bearer
- B3 Room-кэш + repository-слой
- B4 экран «Ученики» (список) + создание теневого ученика и приглашение (share-ссылка)
- B5 экран профиля ученика (цели, сессии — просмотр)
- B6 экран карточки тренировки (просмотр)

**Epic 2 — диктофон (фоновая запись)** ← ядро
- C1 foreground service записи (`microphone`) + разрешения + AAC/m4a
- C2 UI-флоу записи: выбрал тренировку → Запись → Стоп, индикатор в шторке
- C3 WorkManager-заливка в конвейер (upload-url → PUT → transcribe)
- C4 устойчивость: резюмирование при обрыве сети, локальное хранение до подтверждения
- C5 экран статусов: запись → заливка → расшифровка → инсайты
- C6 E2E: 60+ мин в кармане с локом → инсайты видны (тренеру в app, ученику в вебе)

**Epic 3 — разбор тренировки**
- D1 экран транскрипта (просмотр, скачать)
- D2 вставка инсайтов (paste-from-Claude) + авто-генерация (generateAiInsights)
- D3 ревью карточек: approve / reject / edit (title/body/tag)
- D4 переупорядочивание карточек (drag-and-drop)
- D5 завершить ревью → уведомление ученику (Telegram/FCM)

**Epic 4 — ведение (паритет тренера с вебом)**
- E1 создать/править тренировку: упражнения + навыки (поиск по справочникам)
- E2 создать цель для ученика
- E3 управление учениками: правка профиля, переиздать/отозвать приглашение
- E4 библиотека карточек-шаблонов: коллекции, применить к ученику/сессии
- E5 мастер-план ученика: секции и пункты (создание/удаление)
- E6 справочники: навыки и упражнения (поиск/создание)

**Epic 5 — установка на телефон (APK) + полировка** _(без Play Store)_
- G1 сборка устанавливаемого APK + CI (GitHub Actions выкладывает APK артефактом) — ставим сайдлоадом
- G2 FCM-пуши (новые инсайты, напоминания, завершён разбор)
- G3 оффлайн-режим чтения + состояния загрузки/ошибки/пустоты
- G4 иконка, splash, брендинг (низкий приоритет)
- G5 финальная приёмка всего тренерского флоу на реальном телефоне из установленного APK

> **Play Store исключён по решению:** страница магазина, закрытый бета-трек и требования магазина не
> делаем — приложение ставится напрямую APK на личный телефон.

## Порядок и параллельность

- **Старт сразу (параллельно):** A1, A2 (бэкенд) и B1 (scaffold приложения) — независимы.
- A3/A4/A5 — после A1+A2 (можно тремя агентами параллельно по доменам).
- B2 — после A2; B3 — после B1; затем B4/B5/B6 параллельно.
- Epic 2 (диктофон) — после B-каркаса и A4; внутри: C1 → (C2‖C3) → C4/C5 → C6.
- Epic 3 — после A5 read/write и B6; задачи D1–D4 во многом параллельны.
- Epic 4 — после A5; E1–E6 параллельны.
- Epic 5 — после ядра (1–3); G-задачи в основном параллельны, G5 последняя. G1 (APK) — раньше, без него нечего ставить.

Задачи помечаются в теле метками «Зависит от: #…» / «Параллельно с: …», чтобы несколько агентов брали
независимые задачи одновременно без конфликтов по файлам.

## Риски и как держим

- Развитие веба ломает API → версионирование `/v1` + контракт-тесты (A6).
- Потеря длинной записи → WorkManager + резюмирование + локальное хранение до подтверждения; увеличенный
  TTL signed URL (A4).
- Политики Google по фоновому микрофону → честный foreground service с видимым уведомлением, корректные
  разрешения (`RECORD_AUDIO`, `FOREGROUND_SERVICE_MICROPHONE`).
- Авторизация через WebView Гречки → строго перехват токена, без хранения чужих кук; Google как fallback.
