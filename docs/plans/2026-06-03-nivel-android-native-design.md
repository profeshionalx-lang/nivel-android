# Nivel — нативное Android-приложение (полный порт)

**Дата:** 2026-06-03
**Статус:** дизайн утверждён, готов к декомпозиции на эпики/issues
**Автор обсуждения:** Edgar + Claude (brainstorming)

---

## Зачем

Боль тренера: после тренировки нет времени вручную заливать аудиозапись в веб-приложение. Нужно
записывать прямо на телефоне (в кармане, с заблокированным экраном, 60–90 минут) и чтобы запись
автоматически уходила в существующий конвейер «транскрипция → инсайты».

Браузерный веб этого надёжно не умеет: при блокировке экрана/сворачивании вкладка усыпляется и запись
рвётся. Нативное Android-приложение с **foreground service** (тип `microphone`) пишет в фоне штатно —
это и есть причина строить нативку.

Решено идти не «тонким компаньоном-диктофоном», а **полным нативным портом** всего Nivel на
**Kotlin + Jetpack Compose**.

---

## Ключевой принцип: одна сущность, не две

И веб, и Android ходят в **один и тот же бэкенд** и в **одну и ту же базу Supabase**
(`gqcyaxxhvyvpzuhoysis`). База — единственный источник правды. Приложение и веб — два разных клиента
(окна) к одним данным.

Следствие: всё, что тренер делает из приложения (запись, карточки, цели), мгновенно видно ученику в
вебе — потому что данные общие. Ученикам ничего ставить не нужно; их веб-опыт не меняется. Меняется
только окно ввода тренера.

Бизнес-логика **не дублируется**: существующие server actions (`audio.ts`, `aiInsights.ts`,
`sessions.ts`, dashboard `data.ts` и т.д.) рефакторятся так, чтобы их ядро вызывалось и из web-actions,
и из нового REST API. Android — ещё один клиент того же бэкенда.

## Технические решения (приняты, с обоснованием)

| Решение | Что выбрано | Почему |
|---|---|---|
| **Клиент** | Kotlin + Jetpack Compose | Современный нативный стандарт Android, лучшая поддержка, полный доступ к фоновой записи |
| **Репозиторий** | Отдельный `nivel-android` | Разные тулчейны (Gradle vs npm); чистая HTTP-граница здоровее монорепо; своя CI и доска задач |
| **Граница с бэкендом** | REST API `/api/v1/*` (JSON) в текущем NIVEL-репо | Server actions нельзя вызвать из нативки; нужен явный HTTP-контракт |
| **Переиспользование логики** | Рефактор ядра actions в shared-функции, поверх — и web-actions, и API-хендлеры | Одна логика, ноль расхождений между вебом и приложением |
| **Авторизация** | Как в вебе: Гречка (через WebView ловим Firebase ID token) + Google Sign-In (нативно, fallback) → обмен на bearer-JWT | Сохраняем привычный вход; `getSession` учим читать и куку, и `Authorization: Bearer` (тот же HMAC-секрет) |
| **Фоновая запись** | Android foreground service, тип `microphone` + `AudioRecord`/`MediaRecorder`, формат AAC/m4a | Надёжная запись в кармане с локом; m4a уже разрешён в существующем конвейере |
| **Надёжная заливка** | `WorkManager` (фоновая дозагрузка, переживает закрытие приложения и сеть) + резюмируемая загрузка чанками | Запись на 90 мин не должна теряться при плохой сети |
| **Локальные данные** | Room (кэш чтения) + repository pattern; источник правды — Supabase | Скорость, оффлайн-просмотр; не вторая база, а кэш |
| **Сеть/архитектура** | Retrofit + OkHttp, Coroutines/Flow, Hilt (DI), DataStore (хранение токена) | Стандартный современный Android-стек |

Версионируем API (`/v1`) и покрываем контракт-тестами, чтобы развитие веба не ломало приложение.

## UX-решения (от тренера)

- **Старт записи:** тренер сначала выбирает ученика/тренировку, затем жмёт «Запись». Запись сразу
  привязана к нужной сессии — не нужно потом гадать, к кому она относится.

## Этапы (роадмап)

### Этап 0 — Фундамент бэкенда *(невидимый пользователю)*
Подготовить бэкенд, чтобы приложение могло безопасно работать с данными.
- Рефактор ядра server actions в переиспользуемые функции.
- REST API `/api/v1/*` поверх этого ядра.
- Bearer-авторизация: endpoint обмена Firebase ID token → HMAC-JWT; `getSession` читает куку ИЛИ bearer.
- Контракт-тесты API.

**Acceptance:** внешний HTTP-клиент с валидным bearer-токеном может прочитать дашборд/сессии/инсайты и
запустить заливку аудио — теми же данными, что видит веб.

### Этап 1 — Приложение, вход, просмотр
- Scaffold Android-проекта (`nivel-android`), Compose, навигация, DI.
- Экран входа: WebView-флоу Гречки + нативный Google fallback → сохранение bearer-токена.
- Просмотр (read-only): дашборд, список тренировок, карточка тренировки, инсайты, ученики.
- Room-кэш + repository.

**Acceptance:** тренер ставит приложение, входит через Гречку, видит свои реальные данные из Supabase.

### Этап 2 — Диктофон *(ядро, ради чего всё)*
- Foreground service записи (тип `microphone`), индикатор «идёт запись» в шторке.
- Флоу: выбрал тренировку → «Запись» → фон/лок → «Стоп».
- Авто-заливка через WorkManager в существующий конвейер (`requestAudioUploadUrl` → upload →
  `transcribeSession`), резюмирование при обрыве сети.
- Статусы: запись → заливка → расшифровка → инсайты готовы.

**Acceptance:** запись 60+ мин с телефоном в кармане и заблокированным экраном доходит до Supabase,
расшифровывается, генерит инсайт-карточки; они видны тренеру в приложении и ученику в вебе.

### Этап 3 — Полное редактирование (паритет с вебом)
- Создание/редактирование тренировок, целей, инсайт-карточек.
- Управление учениками (роль trainer).
- Всё, что тренер делает в вебе, доступно в приложении.

**Acceptance:** тренер ведёт всю работу из приложения без захода в веб.

### Этап 4 — Полировка и публикация
- Push-уведомления (переиспользуем Telegram/серверную инфраструктуру уведомлений где уместно).
- Оффлайн-режим чтения, обработка ошибок/краёв.
- Оформление Play Store, закрытый бета-трек для тренера, релиз.

**Acceptance:** приложение установлено из Google Play, проходит реальную тренировку end-to-end.

---

## Порядок и зависимости

Этап 0 → 1 → 2 — строго последовательно (фундамент нужен раньше входа, вход раньше записи).
Этапы 3 и 4 идут после того, как ядро (0–2) доказано на реальной тренировке.

Намеренно: главная боль (диктофон) закрывается уже на Этапе 2; Этапы 3–4 доводят приложение до полной
замены веба для тренера.

## Риски и как держим

- **Развитие веба ломает API** → версионирование `/v1` + контракт-тесты.
- **Потеря длинной записи** (сеть/закрытие) → WorkManager + резюмируемая чанковая заливка + локальное
  хранение файла до подтверждения.
- **Политики Google по фоновому микрофону** → честный foreground-service с видимым уведомлением,
  корректные разрешения (`RECORD_AUDIO`, `FOREGROUND_SERVICE_MICROPHONE`).
- **Авторизация через сторонний WebView (Гречка)** → строго перехват токена, без хранения чужих кук;
  Google как нативный fallback.
