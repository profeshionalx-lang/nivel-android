# Это нативный Android, а не веб

Этот репозиторий — **нативное Android-приложение на Kotlin + Jetpack Compose**. Здесь НЕТ Next.js,
React и server actions. Бизнес-логика и данные живут в бэкенде `profeshionalx-lang/NIVEL` и в общей
базе Supabase; это приложение — клиент, который ходит в REST API `/api/v1/*`.

Перед написанием кода сверяйся с актуальными гайдами Android/Compose/Hilt/WorkManager — API меняются,
не полагайся на память.

# Mobile-first по определению

Это телефонное приложение. Любой экран проектируется под мобильный портрет в первую очередь.

- Тач-зоны ≥ 48dp (Material), не полагайся на мелкие элементы.
- Нижние шиты (ModalBottomSheet) вместо центрированных диалогов.
- Длинные горизонтальные списки — `LazyRow` со snap, а не вертикальный стек, где это карусель.
- Состояния загрузки/ошибки/пустоты — обязательны на каждом экране.

# Одна база — один источник правды

Никогда не создавай в приложении «вторую реальность». Источник правды — Supabase через REST API
бэкенда. Room — это **кэш чтения**, а не отдельная база. Любая запись идёт на сервер; локально только
кэшируем и показываем оптимистично, но сервер решает.

# План — гайдлайн, acceptance — закон

Планы и issue пишутся в момент X, ты выполняешь в момент Y. За это время API-контракт или соседние
экраны могли измениться. **Не пиши код по плану слепо.**

**Перед реализацией:**

1. **Прочитай реальное состояние** файлов, упомянутых в issue, и актуальный контракт API (см. эпик
   «Фундамент» в репо NIVEL — там описаны endpoints `/api/v1/*`).
2. **Сравни с issue.** Совпадение → пиши по плану. Мелкие расхождения → адаптируй, сохрани цель.
   Большие расхождения → реализуй **цель из секции Acceptance** по реальному состоянию.
3. **Не блокируй** задачу из-за устаревшего плана. Блок — только если acceptance физически невыполним
   (нет нужного endpoint, нет доступа, противоречивые требования).
4. **В PR** отдельным блоком `## Отклонения от плана` отметь, если адаптировал.

**Главное правило:** ты отвечаешь за то, чтобы acceptance работал на реальном коде. Acceptance — это
контракт.

# Зависимость от бэкенда

Многие задачи приложения зависят от endpoints, которые делаются в эпике «Фундамент» (репо NIVEL).
Если нужного endpoint ещё нет — это **не блокер по умолчанию**: замокай ответ по согласованному
контракту из дизайн-дока, помечай `// TODO: реальный endpoint #<номер>` и двигай UI. Реальную интеграцию
доделаешь, когда endpoint появится. Если контракта нет вообще — тогда блок.
